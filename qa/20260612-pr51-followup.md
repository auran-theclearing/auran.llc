# PR #51 Follow-up Items

Captured from Envoy review threads that are non-blocking style/design observations.
None of these affect whether the song uploads, analyzes, or plays back correctly.

## 1. Simplify `min(20000, nyquist)` to just `nyquist`

**File:** `memory.py:2255`
**What:** `sr` is hardcoded to 22050, so `nyquist` is always 11025. The `min(20000, nyquist)` is defensive against a future sample rate change that may never come. Could just use `nyquist` directly.
**Impact:** None — produces identical output.

## 2. Mobile Safari Blob download verification

**File:** `index.html:1067-1075`
**What:** The old transcript export had a `window.open()` fallback that worked when `<a download>` didn't trigger on mobile Safari. The new Blob path drops that fallback. `<a download>` + Blob is well-supported on modern iOS Safari but hasn't been tested on Olivia's phone.
**Impact:** If it fails on iOS, transcript export silently does nothing (error now surfaces in status bar, but no fallback download). Test on phone to confirm.

## 3. `librosa.get_duration(path=)` double decode concern

**File:** `memory.py:2235`
**What:** Calling `get_duration(path=)` before `librosa.load()` may decode/probe the file twice. For short files this is negligible. For very long recordings on 0.5 vCPU it could add latency. `soundfile.info()` would be lighter, or could derive truncation from whether the loaded array length equals `max_dur * sr`.
**Impact:** Slight extra latency on long files. Not blocking — current 30s/120s cap means the probe is fast relative to FFT.

## 4. `/transcript/db` endpoint still exists

**File:** `server.py:821`
**What:** The client-side Blob export replaced `POST /transcript`, which was deleted. But `POST /transcript/db` (DB-sourced authoritative transcript with recall blocks and server timestamps) is still there. It's a different endpoint with a different purpose, but worth deciding whether to consolidate on one transcript source long-term.
**Impact:** None — different feature, different data source.

## 5. Cross-time comparison if analyses are ever persisted

**What:** The switch from native sample rate to 22050 Hz means spectral metrics (centroid, bandwidth, band energies) are not comparable to any prior analyses of the same file. No historical analyses are stored in memory yet, so this is hypothetical. If we start persisting frequency results, version the analysis params alongside the data.
**Impact:** None currently — no stored analyses exist.
